~/character encoding

UTF-8 defines over a million characters using one, two, three, or four 8-bit words. These are also called bytes, octets, or code points. The more common characters are encoded using only one byte, making UTF-8 a variable length encoding.

One or more of the first, high, or left-most bits are significant digits. They indicate how many bytes are in the character. In a one-byte character the first, or left-most, bit is set to 0 and the remaining seven bits can have values from 0000000 to 1111111, or 0 to 127 in decimal. These 128 values are enough for, and are backwards-compatible with, US-ASCII.

The name UTF-8 is derived from Unicode (itself derived from Universal Coded Character Set) Transformation Format – 8-bit. Officially it is spelt UTF-8 (caps and hyphen). The name "utf-8" may be used by all IANA standards such as CSS, HTML, XML, and HTTP headers since the declaration is case insensitive. W3C recommends it as the default format for all XML and HTML.

Sometimes the bytes cannot be parsed as UTF-8 and can produce an unhandled exception. However nowadars any unparsable code is replaced with something like "�".

Many Windows programs add a BOM (byte order mark), which is not relevant to byte order and can cause errors. For example in Linux, using the command less on a Windows file may produce the error "may be a binary file. See it anyway?". Less does this because it cannot parse the character set as text. A workaround is to use the --force switch in less.

The default encoding in Unix is UTF-8. You can see which encoding your shell is using with the command locale.

In Linux, a line break is a single character: the Line Feed (LF), represented as 0A in UTF-8 hex. In Windows text files a line break is two characters: a Carriage Return (CR) followed by a Line Feed (LF). These can be cleaned up with the command dos2unix. Dos2unix can also convert encoding such as UTF-16 to UTF-8. iconv converts text from one encoding to another encoding.
